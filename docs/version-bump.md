# Bumping the wrapped Mantine version

Moving the wrapper to a newer Mantine release is a scripted afternoon, not
archaeology. The whole surface — every `@mantine/core` component plus dates,
charts, modals, spotlight, notifications, and every `use*` hook — is generated
from committed inputs under [everything-minus-excludes scope](../codegen/scope.edn),
so a bump is: refresh the inputs, move every pin in one pass, regenerate, run the
verify loop, and read the generated diff.

## The pins (move all of them in one pass)

A version lives in four places. Bump them together so nothing lags:

1. **`package.json` `devDependencies`** — the exact-pinned `@mantine/*` packages
   (`core`, `hooks`, `notifications`, `modals`, `spotlight`, `dates`, `charts`),
   **plus their peer deps `dayjs` and `recharts`** (Mantine's dates/charts pull
   these). These drive the installed `node_modules` the generator enumerates.
2. **`src/main/deps.cljs` `:npm-deps`** — the ranges shipped to consumers
   (shadow-cljs auto-installs them). Every `@mantine/*` range **and the `dayjs` /
   `recharts` ranges**. Keep these as `^`-ranges matching the new floor;
   `react`/`react-dom` stay unlisted (the consuming app owns React).
3. **`codegen/generate.clj` `mantine-version`** — the constant stamped into
   generated docstrings, the `GENERATED by …` header, and namespace docstrings.
4. **Input provenance headers** — `bb extract` rewrites the `;; GENERATED by
   bb extract … (Mantine <v>)` banners in the committed inputs, so refreshing
   (below) keeps them honest automatically.

## Procedure

```sh
# 1. Clone the target tag (shallow is fine) and build its docgen output.
git clone --depth 1 --branch <tag> https://github.com/mantinedev/mantine /tmp/mantine
cd /tmp/mantine && yarn install && yarn tsx scripts/docgen   # ~1 min total
cd -                                                         # back to the wrapper repo

# 2. Refresh the committed generator inputs FROM that clone.
bb extract /tmp/mantine
#   -> codegen/input/docgen.json        (verbatim copy of the docgen output)
#   -> codegen/input/component-docs.edn (descriptions + slugs + polymorphic flags)
#   -> codegen/input/hook-docs.edn      (hook descriptions)

# 3. Bump EVERY pin in one pass (the four locations above):
#    package.json, src/main/deps.cljs (incl. dayjs/recharts), codegen/generate.clj.

# 4. Reinstall so node_modules matches the new pins, then regenerate.
npm install
bb generate

# 5. Run the verify loop (or `bb ci`, which chains all of it).
bb generate                     # run twice: the second run must produce NO git diff (idempotent)
npx shadow-cljs release demo    # "Build completed … 0 warnings" (ignore the JVM Unsafe lines)
node scripts/verify-demo.mjs    # every assertion passes ("ALL … VERIFIED")
bb jvm-load                     # every generated ns JVM-loads; call-time throw names the wrapper
bb coverage                     # scoped docgen entries vs generated defs — 0 missing

# 6. Review the generated diff (see below).
git diff -- src/main/mantine/
```

`bb extract` reads the clone's `apps/mantine.dev/src/.docgen/docgen.json` (hence
the `yarn tsx scripts/docgen` step must run first) and the docs-app MDX data under
`apps/mantine.dev/src/mdx/data/` (`mdx-{core,dates,charts,others,hooks}-data.ts`).
The four-pattern PoC measured the whole clone + docgen at roughly a minute of
compute, so the refresh is a cheap scripted step, not a heavy one.

## Reading the generated diff

Under everything-minus-excludes scope the generated surface is derived, not
curated, so a bump's real payload shows up in `git diff -- src/main/mantine/`.
That is where you review the version's API change, and what to look for:

- **New components / hooks** appear as brand-new `(def …)` forms. Any component
  the new version adds (and exports from a wrapped package + carries in
  `docgen.json`) and any new `use*` export of the `@mantine/hooks` barrel surface
  **automatically** — no scope edit needed.
- **Removed / renamed components** disappear from the diff (a removed export is
  no longer in the universe; the generator logs `SKIP component …` for a name
  that lingers in `docgen.json` but is no longer exported). A rename reads as one
  `def` removed and one added.
- **Changed prop tables** show up inline in the docstrings — new/removed props,
  changed types, defaults, `REQUIRED` flags, and polymorphic status all flow from
  the refreshed `docgen.json` + `component-docs.edn`.
- **`bb coverage`** independently recomputes the scoped surface and fails if any
  intended def silently went missing, so a scope/resolution regression can't hide
  in the diff.

The **exclude list in `codegen/scope.edn` is the escape hatch.** Because scope is
"everything minus excludes", you never add a name to opt a new component/hook *in*
— it arrives on its own. You edit the excludes only to opt something *out*
deliberately: a component that doesn't wrap cleanly, or a companion `use*` hook
that shares another hook's docs page and so carries no standalone description
(the current hook excludes are exactly those doc-less companions). If the bump's
diff shows a new entry you don't want shipped, add it to the relevant `:exclude`
set and regenerate.

## Refreshing the inputs: needed vs. skippable

`bb extract` is cheap and refreshes all three inputs in one pass, so **running it
on every bump is the default.** The committed inputs already carry the full
props surface plus ~150 component and 81 hook descriptions, which makes a skipped
refresh a graceful degradation rather than a break — useful to know when the
clone's toolchain won't cooperate.

A refresh is **needed** (not skippable) when:

- the target version **adds, removes, or renames any component** — `docgen.json`
  gates the component universe, so without a refreshed copy a new component can't
  surface and a removed one keeps generating from stale data; or
- you want **prop tables, descriptions, slugs, or polymorphic flags** to reflect
  the new version (all of these come from the refreshed inputs).

A refresh is **skippable** (the committed inputs are a good-enough fallback) for a
narrow bump that leaves the component surface untouched — e.g. a hook-only change
or a `dayjs`/`recharts`-only bump. Hook defs are aliased from the live
`@mantine/hooks` exports (`node_modules`), so a genuinely new hook still surfaces
from the package bump alone; only its description would be absent, degrading to a
thinner docstring — never a generation failure. In that case the descriptions
already committed carry the rest.
